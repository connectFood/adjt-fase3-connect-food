1) Objetivo da refatoracao
- Reorganizar o servico de payment seguindo arquitetura hexagonal (domain, application, infrastructure).
- Atender requisitos do tech challenge:
  - Consumir pedido.criado.
  - Processar pagamento via Procpag (api de risco externa).
  - Publicar pagamento.aprovado ou pagamento.pendente.
  - Garantir resiliencia com timeout, retry e circuit breaker.
  - Reprocessar pagamentos pendentes.

3) Fluxo funcional implementado
3.1 Fluxo principal (pedido.criado)
1. OrderCreatedConsumer consome evento pedido.criado.
2. ProcessPaymentOnOrderCreatedUseCase aplica idempotencia por orderUuid.
3. Cria transacao PENDING quando necessario.
4. PaymentOrchestratorService executa tentativa de pagamento com resiliencia.
5. Se aprovado:
   - atualiza status para APPROVED
   - publica pagamento.aprovado
6. Se falhar/timeout/indisponibilidade:
   - mantem status PENDING
   - publica pagamento.pendente (apenas no fluxo inicial)

3.2 Fluxo de reprocessamento (pagamento.pendente)
1. PaymentPendingWorker consome pagamento.pendente.
2. ReprocessPendingPaymentUseCase tenta pagamento novamente.
3. Se aprovado:
   - atualiza para APPROVED
   - publica pagamento.aprovado
4. Se continuar pendente:
   - atualiza no banco
   - NAO republica manualmente pagamento.pendente
   - lanca excecao para retry topic

4) Resiliencia aplicada (Procpag)
- Componente: PaymentResilienceExecutor
- Estrategias:
  - TimeLimiter
  - Retry
  - CircuitBreaker
- Parametros lidos de configuracao:
  - resilience.procpag.timeout-ms
  - resilience.procpag.retry.max-attempts
  - resilience.procpag.retry.wait-ms
  - resilience.procpag.circuit-breaker.*

5) Ajuste de problema de loop de mensagens
Problema observado:
- Reprocessamento publicava novas mensagens pendentes a cada falha, inflando o topico.

Correcao aplicada:
- No reprocessamento, nao ha republicacao manual de pagamento.pendente.
- Reentrega futura passou a ser feita por retry topics do Spring Kafka.

6) Retry topic + DLT para pendencias
- Implementado em PaymentPendingWorker com @RetryableTopic + @DltHandler.
- Comportamento:
  - quando pagamento continua pendente, listener lanca excecao;
  - mensagem vai para topicos de retry com backoff;
  - apos esgotar tentativas, vai para DLT.

Configuracoes adicionadas:
- payment.pending-retry.attempts
- payment.pending-retry.delay-ms
- payment.pending-retry.multiplier
- payment.pending-retry.max-delay-ms
- payment.topics.payment-pending-dlt
